FabricGuard Rule Language

Anatomy of an FRL rule

A governance rule is not free text — it has structure a machine can evaluate.
RULE ws-admin-groups-001 {
  NAME:       "Workspace must have admin groups"
  VERSION:    "1.0.0"
  SEVERITY:   ERROR
  APPLIES_TO: Workspace

  PARAMS { admin_groups: List<EntraGroup> }

  CHECK SELF.PERMISSIONS(ADMIN)
        CONTAINS_ALL $admin_groups

  FINDING:     "{displayName} is missing …"
  REMEDIATION: "Add the groups as Admins"
}
APPLIES_TO
What it targets. Any Fabric item type — and any new type Microsoft adds is valid with no language change.
PARAMS
Reusable inputs. Typed parameters referenced as $name, so one rule serves many tenants.
CHECK
The condition. SELF.<path> addresses any object property. Multiple CHECKs all must pass.
SEVERITY
How much it matters. ERROR · WARNING · INFO.
FINDING / REMEDIATION
What to tell the user when it fails, and how to fix it — templated with live values.
Stored verbatim in Elastic alongside parsed metadata  ·  VERSION + is_current make every rule immutable and history-aware
